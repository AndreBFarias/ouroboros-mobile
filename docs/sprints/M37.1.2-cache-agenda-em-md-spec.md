## 1. Achado / motivação

M37.1 introduziu cache de eventos do Google Calendar em arquivo
**JSON único** por pessoa
(`media/cache/agenda-<pessoa>.json`). Quebra a invariante do
projeto "tudo o que o usuário vê no app é `.md` no Vault, com
binários originais em pastas dedicadas" — auditada em
2026-05-05 com o dono.

Decisão durável (2026-05-05): **migrar para um `.md` por
evento**, alinhando a agenda ao padrão dos demais módulos
(humor, diário, marco, medida, evento, ciclo, exercício,
alarme, tarefa, contador). Vantagens:

- **Syncthing-friendly**: cada evento é um arquivo pequeno
  independente — sincing seletivo, conflitos isolados, sem
  reescrita de blob inteiro a cada mudança remota.
- **Legibilidade**: usuário pode abrir um `.md` num editor e
  ler a agenda sem o app rodando.
- **Consistência**: deleta-se a exceção; passa a existir 1
  padrão único de persistência para "registros do usuário".
- **Backup/restore**: o ZIP M-EXPORT-COMPLETO já trata `.md`
  byte-a-byte; agenda entra natural.

## 2. Path canônico

Raiz dedicada `agenda/`, agrupada por pessoa para permitir
sincing seletivo:

```
agenda/<pessoa>/<YYYY-MM-DD>-<eventId>.md
```

Exemplo:
```
agenda/pessoa_a/2026-05-07-abc123def456ghi789jkl012mn.md
agenda/pessoa_b/2026-05-08-xyz987uvw654rst321qpn.md
```

`<eventId>` vem direto da resposta do Google Calendar API (formato
Base32hex, ~26 chars, slug-safe). `<YYYY-MM-DD>` é o `inicio` do
evento (data local), facilitando ordenação alfabética e listagem
por dia.

## 3. Frontmatter schema (zod)

`AgendaEventoSchema` em `src/lib/vault/agenda.ts`:

```ts
export const AgendaEventoSchema = z.object({
  id: z.string().min(1),
  pessoa: z.enum(['pessoa_a', 'pessoa_b']),
  titulo: z.string().min(1),
  inicio: z.string().datetime({ offset: true }),  // ISO com TZ
  fim: z.string().datetime({ offset: true }),
  local: z.string().optional(),
  fonte: z.literal('google_calendar'),
  sincronizado_em: z.string().datetime({ offset: true }),
});
export type AgendaEvento = z.infer<typeof AgendaEventoSchema>;
```

Body do `.md`: descrição do evento (campo `description` da API
do Google, opcional, em markdown).

Exemplo de arquivo:

```markdown
---
id: abc123def456ghi789jkl012mn
pessoa: pessoa_a
titulo: Reunião com equipe
inicio: 2026-05-07T13:00:00-03:00
fim: 2026-05-07T14:00:00-03:00
local: São Paulo
fonte: google_calendar
sincronizado_em: 2026-05-05T20:30:00-03:00
---

Reunião semanal de status. Pauta no link do convite.
```

## 10. Decisões tomadas

- **Path `agenda/<pessoa>/...md`** (raiz dedicada, não
  `inbox/`): inbox semanticamente é "captura recente do
  usuário"; agenda é espelho remoto. Raízes dedicadas
  pré-existentes (`marcos/`, `medidas/`, `tarefas/`) cobrem
  esse caso.
- **`<YYYY-MM-DD>-<eventId>.md`** (data prefixada): ordenação
  alfabética = ordenação cronológica; facilita listagem de dia
  via `startsWith` sem ler frontmatter.
- **`sincronizarSnapshotAgenda` em vez de
  `set/delete` granular**: API mais simples; matches o uso real
  ("acabei de buscar do Google, atualizar o estado local").
  Diff por `sincronizado_em` em vez de id-set evita manter
  cursor externo.
- **Boot hook de migração**: usuários que abriram o app entre
  M37.1 (commit `91710ab`) e M37.1.2 (esta) podem ter
  `media/cache/agenda-*.json` no Vault. Migração silenciosa
  preserva dados.
- **Caches de humor + finanças continuam JSON**: são agregações
  readonly geradas pelo backend Python — não são "registro do
  usuário" criado no app, são pré-computações de leitura.
  Exceção legítima documentada na auditoria 2026-05-05.
- **Sem mudança de UX**: usuário não vê diferença visual ou
  comportamental. Apenas arquitetura interna.
- **`AgendaEvento.fonte = 'google_calendar'`** (literal por
  enquanto, vai virar `z.enum(['google_calendar', ...])` se
  M37.x suportar mais provedores no futuro).
