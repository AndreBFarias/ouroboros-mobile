Substituir o stub `scripts/build-mockups.mjs` por uma toolchain real
de regeneração JSX→HTML estático para os mockups de design das 22
telas core (Telas 01–28 namespace JSX). Saída: arquivo único

## 4. Restrições

- **Regra −1** (Anonimato): zero referência a IA em qualquer
  arquivo novo de `scripts/mockups/`. JSX-fonte original já está
  conforme; não reintroduzir nomes próprios.
- **Sem emojis** em código, docs ou commits.
- **PT-BR completo**: nomes de variáveis e funções em
  `scripts/mockups/*.mjs` em **inglês camelCase** (convenção
  build tooling); strings de log em PT-BR Sentence case sem acento
  (convenção shell — alinhado com BRIEF §1.4 sobre commits).
  Documentação em `docs/MOCKUPS-INVENTARIO.md` e
  `docs/CONTEXTO.md` em PT-BR completo com acentuação.
- **TypeScript strict**: scripts são `.mjs` puro Node, não
  passam por `tsc`. Não introduzir `.ts` em `scripts/mockups/`
  para não complicar o pipeline de build.
- **Não tocar runtime do app**: zero alteração em `src/`, `app/`,
  `tests/`. Esta sprint é tooling de docs.
- **Não interferir com bundle frozen**: o arquivo
  `docs/Ouroboros_24_telas-standalone.html` continua versionado e
  intacto. O rebuild fica em arquivo separado para permitir diff
  visual e rollback trivial.
- **Não exigir rede** em runtime do build: dependências React via
  esm.sh ficam **só no HTML gerado** (browser baixa em runtime),
  não no Node durante `npm run build-mockups`. Build em si usa só
  `esbuild` local. Aceitável porque mockups são abertos em browser
  com internet; offline-first não é requisito desta toolchain.
- **Não rodar build em CI** desta sprint: adicionar comando ao
  `package.json scripts` mas **não** plugar em
  `scripts/smoke.sh`. Justificativa: o output é arte estática,
  não código que quebra com regressão silenciosa; e build com
  esbuild adiciona ~1-3s ao smoke já tenso (~30s).
- **Limites de escopo (não-objetivos):**
  - Não regenerar pan/zoom canvas (`design-canvas.jsx` 620 linhas)
    — fica para sprint futura M19.x.3 se houver demanda.
  - Não fundir bundle de 22 telas com bundle de Tela 25/26
    namespace M00.6 — namespaces continuam separados.
  - Não renomear arquivos de bundle (`Ouroboros_24_telas...`
    permanece com nome atual; renomeação coordenada fica para
    sprint futura quando alguém quiser unificar).
  - Não reproduzir post-its decorativos amarelos do canvas
    (`DCNote`, `PostIt` etc).
  - Não reproduzir drag-to-reorder de artboards.
  - Não tocar `Ouroboros_telas_25_26-standalone.html` (manual,
    HTML/CSS escrito à mão; toolchain JSX→HTML não cobre porque
    não há JSX-fonte para essas telas).

   global usado pelos JSX-fonte.
4. **Criar `scripts/mockups/build.mjs`.** Pseudo-código:
   ```js
   import esbuild from 'esbuild';
   import fs from 'node:fs/promises';
   import path from 'node:path';

   const ROOT = path.resolve('docs/design-canvas-export/project');
   const OUT = path.resolve('docs/Ouroboros_24_telas-standalone-rebuild.html');
   const SHIMS = path.resolve('scripts/mockups/dc-shims.mjs');
   const TEMPLATE = path.resolve('scripts/mockups/template.html');

   // 1. esbuild bundle: shims + primitives + secao-{a..e}.jsx em iife.
   const bundleResult = await esbuild.build({
     entryPoints: ['scripts/mockups/entry.mjs'],  // gerado em runtime
     bundle: true, format: 'iife', loader: { '.jsx': 'jsx' },
     jsx: 'transform', jsxFactory: 'React.createElement',
     write: false, target: 'es2020',
     external: ['react', 'react-dom', 'react-dom/client'],
   });
   const bundleJs = bundleResult.outputFiles[0].text;

   // 2. ler theme.css.
   const themeCss = await fs.readFile(path.join(ROOT, 'theme.css'), 'utf8');

   // 3. injetar no template HTML.
   const template = await fs.readFile(TEMPLATE, 'utf8');
   const html = template
     .replace('/* THEME_CSS */', themeCss)
     .replace('/* BUNDLE_JS */', bundleJs);

   // 4. gravar.
   await fs.writeFile(OUT, html);
   console.log(`build-mockups ok: ${OUT} (${html.length} bytes)`);
   ```
   `entry.mjs` é gerado em runtime (ou estático) com:
   ```js
   import './dc-shims.mjs';
   import '../../docs/design-canvas-export/project/primitives.jsx';
   import '../../docs/design-canvas-export/project/secao-a.jsx';
   import '../../docs/design-canvas-export/project/secao-b.jsx';
   import '../../docs/design-canvas-export/project/secao-c.jsx';
   import '../../docs/design-canvas-export/project/secao-d.jsx';
   import '../../docs/design-canvas-export/project/secao-e.jsx';
   ```
   Como cada seção termina com `Object.assign(window, {...})` ou
   chama componentes inline, o entry.mjs também precisa montar
   uma árvore React `<App>` que renderize cada seção em
   sequência. Ler topo de cada `secao-{a..e}.jsx` para descobrir
   se há um componente raíz exportado; se não houver, criar um
   no entry com `React.createElement` chamando os render
   helpers identificados.

## 7. Decisões tomadas

- **Caminho A (replicar primitivos + esbuild + react-dom)
  escolhido sobre B e C.** Investigação revelou que 11 dos 13
  primitivos usados pelas seções já estão definidos em
  `primitives.jsx` (linha 28 confirma `Object.assign(window,
  {...})`). Apenas `DCSection` e `DCArtboard` precisam de shim
  render-only — trivial em ~30 linhas. Caminho B (reescrever
  JSX-fonte) descartado: tocaria 5 arquivos de 397 a 609 linhas
  cada, alto risco de quebrar paridade visual. Caminho C
  (Storybook) descartado: tooling pesado para docs estáticas
  apenas.
- **Saída em arquivo separado, não substitui bundle frozen.**
  `docs/Ouroboros_24_telas-standalone-rebuild.html` coexiste com
  `docs/Ouroboros_24_telas-standalone.html` (frozen). Permite
  diff visual durante validação e rollback trivial. Quando
  paridade visual for validada em sprint futura,
  `docs/Ouroboros_24_telas-standalone-rebuild.html` pode
  substituir o frozen via simples `mv`. Esta sprint **não**
  promove a substituição — fica como tarefa explícita de uma
  M19.x.2 ou ato manual do dono.
- **Sem pan/zoom canvas (`design-canvas.jsx`).** O canvas
  oferece valor visual em ambiente de design tooling externo;
  em HTML estático local, scroll vertical do navegador cobre o
  caso de uso. Reduz escopo da sprint em ~620 linhas de JSX
  complexo (drag-reorder, focus overlay, sidecar JSON).
- **JetBrains Mono via Google Fonts CDN, não local.** Bundle
  frozen depende de fonts CDN ou fallback. Reproduzir o mesmo
  comportamento mantém paridade. Fonts locais em `public/fonts/`
  ficam pelo BRIEF A23 só para o app; bundle estático de docs
  pode pagar custo de rede uma vez.
- **Estimativa elevada de 1,5-3h para 2,5-4h.** Razões:
  1. JSX-fonte usa pattern global (`window.PhoneFrame =...`) que
     exige entry.mjs custom (~30min extra).
  2. Scan de cada secao-{a..e}.jsx para identificar root render
     helper precisa cuidado (~30min extra).
  3. Validação de paridade visual com bundle frozen é manual e
     subjetiva (~60min extra).
  Se executor estourar 3,5h, dividir:
  - **M19.x.1**: build.mjs + dc-shims.mjs + bundle gerando
    HTML que abre sem erro (não cobra paridade visual).
  - **M19.x.2**: paridade visual + checkpoint comparativo +
    docs.
- **Build não rodando em CI.** Adicionar `npm run build-mockups`
  em `scripts/smoke.sh` ou `hooks/pre-commit` adicionaria 1-3s
  ao smoke já em ~30s. Output é arte estática, não código que
  quebra silenciosamente. Decisão pragmática: rodar manualmente
  quando JSX-fonte for editado.
- **Não unificar bundle de 22 telas com bundle de Tela 25/26
  namespace M00.6.** Continua o split documentado em
  `docs/MOCKUPS-INVENTARIO.md` §4. Unificar exigiria reescrever
  Tela 25/26 manuais em JSX-fonte equivalente — fora do escopo
  PARALELO-E.
- **Bundle Hermes Mobile não impactado.** Sprint exclusivamente
  toolchain de docs/HTML; runtime do app intocado.

## 9. Riscos e não-objetivos

### Riscos

- **R1 (médio).** `secao-{a..e}.jsx` podem ter código de top-level
  com side effects que dependem de DOM existir antes do bundle
  rodar. Mitigação: entry.mjs envolve a árvore num `useEffect`
  ou `DOMContentLoaded`. Se quebrar, debug pontual.
- **R2 (baixo).** esbuild ESM com `format: 'iife'` pode falhar
  em transformar JSX se `loader: { '.jsx': 'jsx' }` não estiver
  configurado. Mitigação: testar com um JSX trivial primeiro
  (`scripts/mockups/test-jsx.mjs`) antes de plugar os
  secao-{a..e}.
- **R3 (baixo).** JetBrains Mono via CDN bloqueia render se
  rede ruim. Mitigação: fallback `monospace` em CSS.
- **R4 (médio).** Paridade visual pode ser pior que esperado
  (ex: spacing diferente, post-its ausentes destoam visualmente).
  Mitigação: documentar tudo em
  `M19.x-checkpoint-toolchain.md` como "divergência aceita" e
  oferecer M19.x.3 como sprint de polish.
