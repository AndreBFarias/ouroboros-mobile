## 1. Bug observado

Em `src/components/brand/OuroborosLoader.tsx`:
- `useAnimatedProps` retorna `{ transform: \`rotate(${rotacao.value} 160 160)\` }`.
- `<AnimatedG animatedProps={...}/>` aplica essa prop.

Comportamento web (medido via playwright em /humor):
- t=500ms, t=750ms, t=1000ms: atributo `transform` permanece `"rotate(0 160 160)"`.
- Animação não roda.

Evidência: `docs/validacao-gauntlet-2026-05-03/screenshots/M25.1/A-fail-rotate-zero.png` + relatório.

## 2. Hipóteses (ordem de probabilidade)

1. **`react-native-svg` web não escuta `transform` via `useAnimatedProps`** — o adapter web do `<G>` deserializa apenas a primeira prop e ignora atualizações subsequentes do worklet.
2. **Reanimated 4 web** — em alguns casos exige polyfill `__reanimatedWorkletInit` no bundle; pode estar no Android mas faltando no web (Metro splits).
3. **`AnimatedG` é `Animated.createAnimatedComponent(G)`** — versão de `react-native-svg` 14+ recomenda usar `<G transform={...}/>` sem `Animated`, ou aplicar transform via style num wrapper.

## 3. Fix proposto

Trocar abordagem em `src/components/brand/OuroborosLoader.tsx`:
- Em vez de `AnimatedG` com `useAnimatedProps`, envolver cada anel com `<Animated.View style={{ transform: [{ rotateZ: rotacao.value + 'deg' }] }}>` por fora do `<Svg>`.
- OU usar a abordagem nativa do `react-native-svg` para web: aplicar transform inline no atributo via `useAnimatedReaction` que escreve direto no DOM via ref.
