## 1. Achado / motivação

Bug reproduzido no Gauntlet web em 2026-05-05 21:00 pelo dono:
clicar **"Conectar conta Google"** em `/agenda` trava o app no
estado `offline` com banner "Sem conexão. Mostrando eventos do
cache." em vez de ir para `online`.

**Causa raiz:** `useGoogleAuth.autenticar()` tem branch
`__DEV__ && Platform.OS === 'web'` (`isMockMode()`) que injeta
um token sintético `'mock-access-token-dev-web'` (M37.1
`googleAuth.ts:101`). Mas `calendarApi.listarEventos()` **não
tem branch mock** — chama `fetch` real contra
`googleapis.com/calendar/v3/...` com o token fake. Resposta:
401 (token inválido) ou CORS/rejeição → `ApiError(code='rede')`
→ `setEstado('offline')` em `app/agenda.tsx:104-106`.

Consequência: validação visual end-to-end via fluxo real
("Conectar conta Google" → ver eventos) é impossível no
Gauntlet web. Os checkpoints anteriores (M37.1, M37.1.1,
M37.1.2) usaram **seed direto** via `__gauntlet` ou injeção
de cache mock — não exercitaram o caminho do botão "Conectar".

Bug é **pré-existente desde M37.1** (entregue 2026-05-05
tarde), não regressão de M37.1.2. Sprint corretiva.

## 2. Tarefa

Em `src/lib/services/calendarApi.ts`, na função `listarEventos`,
adicionar branch idêntico ao de `googleAuth.ts`:

```ts
function isMockMode(): boolean {
  const dev = typeof __DEV__ !== 'undefined' && __DEV__ === true;
  return dev && Platform.OS === 'web';
}

export async function listarEventos(
  token: string,
  de: Date,
  ate: Date,
  pessoa: PessoaAutor,
): Promise<EventoCalendar[]> {
  if (isMockMode() && token.startsWith('mock-')) {
    return eventosMockados(de, ate, pessoa);
  }
  // ... lógica fetch real existente, intacta ...
}
```

Função `eventosMockados(de, ate, pessoa)` retorna 5 eventos
deterministicos distribuídos a cada 2 dias a partir de `de`,
com IDs estáveis (para idempotência da migração M37.1.2 e da
sincronização cache via `sincronizarSnapshotAgenda`):

```ts
function eventosMockados(de: Date, ate: Date, pessoa: PessoaAutor): EventoCalendar[] {
  const base = de.getTime();
  return [0, 2, 4, 6, 8].map((deltaDias, idx) => {
    const inicio = new Date(base + deltaDias * 86400_000);
    const fim = new Date(inicio.getTime() + 3600_000);
    return {
      id: `mock-${pessoa}-${idx}`,
      titulo: ['Reunião com equipe', 'Almoço', 'Consulta', 'Treino', 'Estudo'][idx],
      inicio: inicio.toISOString(),
      fim: fim.toISOString(),
      local: idx === 0 ? 'São Paulo' : undefined,
      descricao: idx === 0 ? 'Reunião semanal de status. Pauta no link do convite.' : undefined,
    };
  });
}
```

## 8. Decisões tomadas

- **Mock determinístico com IDs estáveis** (`mock-<pessoa>-<idx>`):
  permite rodar fluxo 2x sem duplicar em Vault, e mantém a
  idempotência verificada em M37.1.2 funcionando.
- **5 eventos a cada 2 dias** (em vez de 30, 90, etc): suficiente
  para validar visualmente sem encher o calendar; alinha com a
  expectativa visual de `A-agenda-mes-com-dots.png` (dots em
  5/7/11/13/15).
- **Branch `token.startsWith('mock-')`** (em vez de só
  `isMockMode()`): defesa em profundidade — se algum bug fizer
  o app rodar com token real em web dev, ainda tenta fetch
  real. Apenas tokens claramente mockados disparam mock.
- **Sem extrair `isMockMode` para helper compartilhado**: hoje
  está duplicado entre `googleAuth.ts` e `calendarApi.ts`. 2
  ocorrências = ainda não vale extração (regra de 3). Se
  M37.2 precisar do mesmo, extrair em
  `src/lib/services/devMockMode.ts`.
- **Sem mock de erros** (401/quota/rate-limit/5xx) em web
  dev: o objetivo aqui é validar fluxo feliz; testes
  unitários cobrem caminhos de erro com fetch mockado de outra
  forma.
