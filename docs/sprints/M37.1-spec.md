## 1. Objetivo

Permitir que o usuário (e o casal) conecte sua conta Google e
**visualize** agenda dos próximos 30 dias na rota raiz `/agenda`.

arquivo no Vault (vide A20 do BRIEF). Sem servidor próprio. Sem
escrita ainda — M37.2 entrega criar/deletar evento.

## 4. Restrições

- **Regra −1**: zero IA, zero nomes reais hardcoded. Email do
  usuário vem da resposta OAuth (runtime).
- **Privacidade absoluta**: tokens só em SecureStore local. Sem
  servidor próprio, sem analytics, sem proxy próprio. Documentado
  em ADR-0007 (parágrafo OAuth ampliado por ADR-0018 nesta sprint).
- Sentence case + acentuação PT-BR.
- TS strict.
- **Escopo OAuth M37.1: `https://www.googleapis.com/auth/calendar.events.readonly`**
  (somente leitura). M37.2 vai pedir reconsentimento subindo para
  `calendar.events` (read+write).
- **Armadilha A21 (BRIEF §4)**: split obrigatório de clientId
  (Expo Go usa proxy; dev-client/release usa custom-scheme).
  Detectar ambiente via `Constants.appOwnership` e selecionar via
  env var `EXPO_PUBLIC_GOOGLE_CLIENT_ID_WEB` ou `_ANDROID`.
- **Armadilha A20 (BRIEF §4)**: cache de eventos NUNCA vai para
  SecureStore (limite ~2KB). Sempre em arquivo no Vault.
- **Cache TTL 1h** com fallback a "stale-while-revalidate" se
  offline.
- **Validação Nível A (Chrome web) NÃO cobre OAuth real** —
  redirect custom-scheme não existe em browser. Plano de
  contorno: em web + `__DEV__`, o store injeta token sintético
  para validar render dos 5 estados visualmente; OAuth real fica
  para Nível B (emulador) ou Nível C (celular).

## 9. Decisões tomadas

- **Split em M37.1 (leitura) + M37.2 (escrita)** confirmado:
  reduz risco de PR enorme, valida refresh/cache antes de incluir
  mutação remota, leitura entrega valor independente.
- **Escopo `calendar.events.readonly` em M37.1** (vs `calendar`
  full): princípio de menor privilégio. M37.2 vai pedir
  reconsentimento subindo para `calendar.events` (Google obriga
  re-prompt em escalada de escopo).
- **Split clientId Expo Go (proxy `auth.expo.io`) vs
  dev-client/release (custom-scheme `ouroboros://`)**: identificado
  por agente de teste (vide A21). Detecção via
  `Constants.appOwnership`. Env vars
  `EXPO_PUBLIC_GOOGLE_CLIENT_ID_WEB` e `_ANDROID` separadas. Sem
  isso, APK quebra em `redirect_uri_mismatch`.
- **Cache em arquivo `media/cache/agenda-<pessoa>.json`, NÃO
  SecureStore** (vide A20): 30 dias de eventos podem estourar o
  limite ~2KB do EncryptedSharedPreferences. Arquivo no Vault
  ainda fica criptografado em repouso (depende do device) e tem
  bonus de aparecer no desktop via Syncthing.
- **Estado `invalido` (refresh token revogado externamente)**
  explícito como banner: usuário sabe imediatamente que precisa
  reconectar; não fica preso em loading silencioso.
- **`react-native-calendars`** (vs custom): biblioteca madura,
  funciona com Reanimated 4 (validar em peer). Tema Dracula
  injetado via `theme` prop.
- **Detecção offline via `@react-native-community/netinfo`**:
  banner "Mostrando cache" quando offline; UI normal com cache
  stale acessível.
- **Validação Nível A com mock OAuth**: branch
  `__DEV__ && Platform.OS === 'web'` injeta token sintético no
  store. Permite testar visualmente os 5 estados sem rede. OAuth
  real só em Nível B/C.
- **2 contas Google (pessoa_a + pessoa_b)**: cada um do casal
  autentica separadamente. Modo "Casal" no chip pessoa une as
  duas agendas (lista combinada com `<PessoaBadge>` por evento).
- **Tratamento explícito de erros HTTP** em `calendarApi.ts`:
  401 → invalido, 403 quota, 429 backoff exponencial,
  5xx retry 1x.
- **ADR-0018 documenta tudo**: split clientId, cache em arquivo,
  ausência de servidor, escopo mínimo.
