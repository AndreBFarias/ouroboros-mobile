## Resolução interina

Tentativas de fix em 2026-05-03:

1. **Latch em escopo de módulo (`let _fontesJaCarregaramAlgumaVez`)**
   — gerou `Maximum update depth exceeded` em React 19 strict mode.
   Reverter.
2. **Combinação `useRef` + `sessionStorage`** — leitura/escrita
   durante render também disparou loop. Reverter.
3. **`useRef` original (M27.1)** — funciona em rota `/_dev/gauntlet`
   sem oscilação observável. Em outras rotas o boot inicial demora
   30-60s pelo bug conhecido do `useFonts` SDK 54 web (já documentado
   como aceito).

Conclusão: M27.1 estabiliza após primeira passagem. A "oscilação"
reportada no manual de 2026-05-03 era predominantemente a primeira
passagem do `useFonts` que demora ~30-60s no Chrome dev. Após esse
tempo, não há piscar.

Reabrir como **M27.3** se a oscilação persistir após sessão
estabilizada (>1min). Solução completa exigirá:
- Suspense boundary com `<SplashScreen>` gerenciado por
  `expo-splash-screen` mantido até `useFonts === true` E
  `useHasHydrated` de todas as stores.
- Migrar `useFonts` para `Font.loadAsync` chamado uma vez fora do
  render tree.

## 1. Bug observado

Após fix M27.1 (useRef `fontesPersistentementeCarregadas` para não re-disparar boot), o loader Ouroboros:
- Aparece corretamente na primeira passada.
- Desaparece quando fontes carregam (~30-45s SDK 54 web).
- **Volta a aparecer** após o conteúdo da rota já estar montado, em ~200-500ms depois.

Reprodução consistente em:
- `/`
- `/humor`
- `/settings`
- `/settings/editar-pessoa`

Em `/_dev/gauntlet` o boot resolve rápido e não oscila — sugere que o gate problemático é específico de rotas não-dev.

Evidência: `docs/validacao-gauntlet-2026-05-03/screenshots/M27.1/A-boot-oscila-em-humor.png` + relatório M-REVALIDACAO-M20-M28.

## 2. Hipóteses (ordem de probabilidade)

1. **Segundo gate de sessão re-liga `mostrarBootScreen`** — `SessaoBootGate` (em `src/lib/boot/`) tem condição que volta a `true` após reidratação assíncrona da store.
2. **`BiometriaGate`** — depende de `SecureStore` async no Android, mock no web; pode retornar `pending` após primeiro render quando algum efeito reseta.
3. **Re-render por mudança de rota interna** — `expo-router` re-monta o gate ao trocar de rota dentro do mesmo bundle.

## 3. Investigação prévia

Antes de qualquer fix:
1. `rg "mostrarBootScreen|setBootScreen|bootScreen" src/ app/` — mapear TODOS os pontos que mexem no estado.
2. Adicionar `console.log` temporário em cada setter durante 1 navegação fresh para sequenciar transições.
3. Confirmar se a oscilação é causada por (a) mesmo state setter sendo chamado de novo, (b) outro state setter que cobre a tela com loader, (c) `@react-navigation` desmontando e remontando o gate.

## 4. Fix proposto (após investigação confirmar hipótese)

Se hipótese 1 confirmada:
- Aplicar mesma técnica do M27.1 (useRef latch) ao `SessaoBootGate`: uma vez que sessão hidratou e gate liberou, NÃO voltar a fechar nessa montagem.

Se hipótese 2:
- `BiometriaGate` no web SEMPRE retornar `bypass=true` (já tem prop `bypass` adicionada no M-GAUNTLET).

Se hipótese 3:
- Mover gate de sessão para FORA do escopo do `Stack`, no `app/_layout.tsx` raiz, antes do roteador renderizar children.
