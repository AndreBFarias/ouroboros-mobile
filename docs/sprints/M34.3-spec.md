# Sprint M34.3 — FAB verde do MenuCapturaVerde sobrepõe FABs próprios das abas

```
DEPENDE:    M34 fechada
BLOQUEIA:   M-CAPTURA-UNIFICADA (precisa decidir UX antes)
ESTIMATIVA: 1-2h
PRIORIDADE: alta (regressão funcional — abas Marcos e Fotos com FAB
            inacessível por sobreposição)
STATUS:     [todo]
```

## 1. Achado / motivação

Validação 2026-05-04 da aba Marcos via Gauntlet revelou que o
**FAB verde** introduzido por M34 (`MenuCapturaVerde`, FAB
canto inferior direito) **sobrepõe** os FABs próprios das abas:

| Tab | FAB próprio | Posição | Estado |
|---|---|---|---|
| Treinos | `<FAB>` "novo treino" (futuro?) | n/a | n/a |
| Fotos | `<FAB>` "adicionar foto" (M11.1) | (769, 900) 56×56 | **bloqueado** |
| Marcos | `<FAB>` "adicionar marco" (M11) | (769, 900) 56×56 | **bloqueado** |
| MenuCapturaVerde | "abrir menu de captura" | (769, 900) 56×56 | sobrepõe |

Coordenadas batem 1:1. O playwright `click` no FAB de marcos retorna
"intercepts pointer events" — confirma que o FAB verde está acima
no z-order.

Resultado: usuário não consegue tocar em "adicionar marco" nem
"adicionar foto" pelo gesto direto.

## 2. Caminhos possíveis

### A — FAB verde absorve as ações por contexto (preferido)

Em vez de cada tab ter seu próprio FAB roxo, o **FAB verde
unificado** abre menu contextual:

- Tab **Treinos**: menu mostra "Novo treino" + 4 opções de captura
  (Foto/Música/Vídeo/Frase).
- Tab **Fotos**: menu mostra "Adicionar foto" como atalho rápido
  + 3 opções de captura (Música/Vídeo/Frase).
- Tab **Marcos**: menu mostra "Adicionar marco" + 4 opções de
  captura.

Vantagem: 1 FAB por tela, semântica clara (verde = registrar),
elimina FAB duplicado.

Custo: refatorar `MenuCapturaVerde` para receber `extras` por tab.

### B — Manter FABs próprios e mover MenuCapturaVerde

FAB verde só aparece em algum espaço dedicado, não no canto inferior
direito (que é "casa" dos FABs principais). Ex: header da tela
Memórias, ou substituir a posição do FABMenu (esquerda).

Custo: rompe convenção de FAB principal = canto inferior direito.

### C — FAB roxo das tabs domina; MenuCapturaVerde só em Memorias screen-level

`<MenuCapturaVerde>` é renderizado no `<MemoriasScreen>` antes do
swipe de tabs. Solução: empurrá-lo para `app/captura.tsx`
(absorvido pelo M-CAPTURA-UNIFICADA) e deixar cada tab com seu
FAB próprio.

Custo: M-CAPTURA-UNIFICADA já tem essa arquitetura — basta
acelerar.

**Preferência:** **A** se M-CAPTURA-UNIFICADA não for executada
imediatamente; **C** se for.

## 3. Entregáveis (caminho A)

- `MenuCapturaVerde` aceita prop `acoesExtras?: Array<{label, icone,
  onPress}>` que renderiza acima das 4 ações de captura.
- Cada `MemoriasFotosTab`/`MemoriasMarcosTab`/`MemoriasTreinosTab`
  passa sua ação correspondente via `useMemo`.
- FAB próprio de cada tab é REMOVIDO.
- Empty state de cada tab continua tendo botão inline para descoberta
  (M11.1 padrão).
- Caso E2E em `tests/e2e/playwright/m34-3-fab-unificado.e2e.ts`
  validando que tap no FAB verde da aba Marcos abre menu com
  "Adicionar marco" + 4 opções de captura.

## 4. Verificação

- Gauntlet `/memoria` → swipe para Marcos → tap FAB verde → menu
  com 5 itens (1 contextual + 4 captura).
- Mesma coisa em Fotos.
- Sheet de adicionar marco abre normalmente quando tap "Adicionar
  marco" no menu.

## 5. Decisões pendentes

- **Cor do ícone "Adicionar marco"** vs cores dos itens de captura
  (foto/música/vídeo/frase usam verde por convenção): item
  contextual deve ter cor diferente para hierarquia? Ou todos verde
  para coerência? Decidir na execução, com olhar de UI/UX.
- **Quando M-CAPTURA-UNIFICADA executar primeiro**, M34.3 pode ser
  resolvida pelo caminho C ao invés de A. Se isso acontecer, esta
  spec fecha como duplicada.
