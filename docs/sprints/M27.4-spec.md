# Sprint M27.4 — SessaoBootGate "Maximum update depth" em reset+seed+abrir rápido

```
DEPENDE:    M27.3 fechada (boot screen estabilizado)
BLOQUEIA:   nada (pré-existente desde M24, não regressão)
ESTIMATIVA: 1-2h
PRIORIDADE: média (afeta apenas ciclo Gauntlet rápido — não atinge
            usuário real, mas polui validação visual)
STATUS:     [todo]
```

## 1. Achado / motivação

Documentado em STATE.md durante M27.3 (2026-05-04):

> Achado pré-existente: `SessaoBootGate` "Maximum update depth
> exceeded" em cenário `__gauntlet.reset() + seed() + abrir()` em
> sequência rápida (<2s). Pré-existente desde M24 (não regressão de
> M27.3). Cascata `useUltimaRota` + `useHasHydrated` após reset das
> 3 stores. Não bloqueia uso real.

Causa raiz hipotética: `SessaoBootGate` lê `ultimaRota` da store
recém-resetada antes de `useUltimaRota` ignorar o primeiro pathname
(M24.1 fix), causando re-render em cascata enquanto as 3 stores
hidratam.

## 2. Diagnóstico inicial

Hipóteses a verificar antes do fix:

1. `useHasHydrated` das 3 stores (sessao/onboarding/vault) tem
   ordem de resolução não determinística após reset; uma delas
   resolve primeiro e dispara `Redirect` enquanto outra ainda
   reset → loop.
2. `SessaoBootGate` consome `useUltimaRota` que retorna `undefined`
   no primeiro frame pós-reset, depois retorna o valor real do
   `setUltimaRota` posterior — re-render dispara `Redirect` 2x.
3. Reanimated em strict mode amplifica o número de updates por
   frame.

## 3. Caminhos possíveis

**A — Latch persistente do bootstrap:** Após primeiro
`bootCompleto: true` em `useBootStatus` (M27.3), `SessaoBootGate`
ignora futuras mudanças de `useHasHydrated`. Reset não dispara
re-bootstrap.

**B — Debounce de reset:** `aplicarReset` de `gauntlet.ts` aguarda
um frame (`requestAnimationFrame`) antes de re-emitir o estado
para os subscribers. Garante que as 3 stores resetam
sincronizadamente.

**C — Suspense boundary com transition:** envolver
`SessaoBootGate` em `<Suspense fallback={<OuroborosLoader/>}>` e
usar `useTransition` para o `Redirect`. Evita render síncrono em
cascata.

**Preferência:** **A** (mais simples, escopo isolado em
`SessaoBootGate.tsx`).

## 5. Verificação

- `__gauntlet.reset() + seed() + abrir('/memoria')` em <2s não
  produz "Maximum update depth" no console.
- Boot continua < 200ms.
- 0 transições do loader em rotas após reset.

## 6. Decisões tomadas

- Latch persistente é segura — uma vez bootstrapado, não há razão
  para repetir o gate. Reset do gauntlet é dev-only.
- Sem mudança em `useUltimaRota` (M24.1 está estável).
