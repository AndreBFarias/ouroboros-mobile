## Armadilhas descobertas (A32-A37)

> Adicionadas a `VALIDATOR_BRIEF.md` §4 (gitignored — este doc é

### A32 — HyperOS bloqueia `adb install -r`

`INSTALL_FAILED_USER_RESTRICTED: Install canceled by user` mesmo
com "Install via USB" + "Depuração USB (Config segurança)" ATIVAS.
Popup HyperOS de confirmação nunca aparece para alguns APKs (filtragem
por assinatura sem garantia Mi).

**Fix canônico:**
```bash
adb push builds/<apk> /data/local/tmp/app.apk
adb shell pm install -r -t /data/local/tmp/app.apk
```

Funciona porque `pm install` roda como `shell` user (privileged) e
não passa pelo verificador HyperOS de origem do arquivo.

### A33 — `Header.tsx` `w-10` corta right slot

`<View className="w-10">` força 40dp no slot direito do Header,
cortando conteúdo maior (pill com ícone+label).

**Fix canônico:**
```tsx
// Antes: <View className="w-10 items-end justify-center">{right}</View>
// Depois:
<View style={{ minWidth: 40, alignItems: 'flex-end', justifyContent: 'center' }}>
  {right}
</View>
```

Cresce com conteúdo, mantém simetria com slot esquerdo (back 40dp).

### A34 — `Button.tsx` MotiView colapsa flex row complexo

MotiView interno do Button aplica `alignItems: center + justifyContent:
center` e isso colapsa filhos `<View flexDirection: row>` no New Arch
(release Android + web). Sintoma: só renderiza ícone, ignora Text irmão.

**Fix canônico:** para pills/CTAs com layout custom (ícone+label,
badge), usar `Pressable` direto inline (~30 linhas). Button.tsx
reservado para CTAs primários full-width com label string simples.

Exemplo: `BotaoRecap` em `app/index.tsx:35-78`.

### A35 — ADB tap usa coords físicas

Screencap pull vem como PNG ~540x1106 (2.0x scale-down do físico
1080x2400 no Xiaomi 2312DRAABG). ADB `input tap` usa coords físicas.

Calcular: `tap_x = screencap_x * physical_w / screencap_w` (= 2x no
2312DRAABG).

**Solução melhor que tap cego:**
```bash
adb shell uiautomator dump /sdcard/ui.xml
adb pull /sdcard/ui.xml /tmp/ui.xml
grep -oE 'text="[^"]*" .* bounds="[^"]*"' /tmp/ui.xml | head -20
```

Bounds vêm em coords físicas absolutas — usar diretamente.

### A36 — ANR em DevLauncherErrorActivity

App ANR (Application Not Responding) em `DevLauncherErrorActivity`
indica crash JS antes do React errboundary pegar — não aparece toast
de erro. Logcat só mostra transition do WindowManager + `Killing X:
user request after error`.

**Diagnóstico:**
```bash
PID=$(adb shell pidof com.ouroboros.mobile)
adb logcat -d --pid=$PID 2>&1 | grep -iE 'TypeError|Invariant|Cannot|fatal'
```

### A37 — Build EAS dev-client demora 10-25min

Build EAS dev-client APK demora 10-25min em fila Free Tier (compila
nativo + assina com keystore EAS).

**Critério para evitar rebuild:** se só código JS mudou (não plugins
Expo nem packages nativos), reusar `builds/dev-client-*.apk` local

### Bug Q5.1 — Speech-recognition aborto silente (resolvido)

**Causa raiz** (via logcat com `console.warn` instrumentado):

```
[transcribe Q5 START] pt-BR
[transcribe Q5 ERROR EVENT] {"error":"aborted","message":"Speech recognition aborted.","code":-1}
[transcribe Q5 FAIL] Error: erro de voz: aborted
```

Android SpeechRecognizer não consegue compartilhar microfone com
`expo-av Audio.Recording`. A spec Q5 pedia "transcribeStream em
paralelo ao Audio.Recording" — tecnicamente impossível em Android.

**Fix:** dois botões separados (`MicrofoneButton` audio-only +
`TranscreverButton` text-only). Usuário escolhe um por vez, zero
conflito de mic. Visual: cyan Mic / orange Captions lado-a-lado.

### Bug Q5.2 — Transcrição parando após 6-8s (resolvido)

`continuous: false` é default do SpeechRecognizer Android: timeout de
silêncio fecha a sessão automaticamente. Fix: `continuous: true` no
`start()`; caller chama `abort()` no release do botão para encerrar.

### Bug Q6 — `GO_BACK was not handled by any navigator` (resolvido)

Em `app/diario-emocional.tsx`, `router.back()` era chamado **duas
vezes** em sequência:
- Pós save: linha 405 chama back direto
- Sheet close emite `onChange(-1)` → linha 428 chama back novamente

A 2ª chamada falhava porque o stack já não tinha rota anterior.

**Fix:** ref guard `backCalledRef` + helper `goBackOnce()` que protege
contra duplo trigger. Funciona em ambos cenários (save success +
swipe-down close).

### Bug Q0 — OAuth `Error 400 invalid_request` (causa raiz fixada,
pendente teste runtime)

Diagnóstico via agente do navegador (Chrome MCP) auditando Google Cloud Console:

1. **Causa raiz:** scope `https://www.googleapis.com/auth/calendar.events.readonly`
   NÃO estava registrado em "Acesso a dados" do consent screen. Quando
   o app pede um scope não-declarado, Google retorna invalid_request
   com mensagem genérica de "OAuth 2.0 policy".
2. **Discrepância secundária:** `env.json` tinha chave `installed`
   (tipo Desktop) mas o único OAuth client do projeto é Android
   `691237256846-sgqvc50rnut8p5emjq44sjfs6jg9tq5g`. Renomeei chave
   para `android` e atualizei `getClientIdFromEnv()` pra preferir
   `android.client_id`, mantendo `installed` como fallback legado.

**Ações realizadas:**
- Scope `calendar.events.readonly` registrado na consent screen via
  agente do navegador (status: "As mudanças de acesso aos dados foram
  salvas.")
- `env.json` renomeado: `installed` → `android`
- `googleAuthFlow.ts:52` lê `env.android?.client_id ?? env.installed?.client_id`

**Pendente:** revalidar login no celular real (user reportou que
"erro do google segue apitando" após primeiro round — pode ser
propagação Google atrasada ou cache dev-client).

### Limitação OAuth do APK via GitHub Actions

Gradle assembleRelease sem keystore configurado cai em **debug
keystore** (SHA-1 padrão do AOSP). O Android OAuth client cadastrado
no Cloud Console (`E4:49:C8:B3:B4:89:F9:26:69:AA:31:1C:38:81:43:44:D3:7D:B3:8C`)
não bate.

Sintoma: OAuth Google retorna `Error 400 invalid_request` no APK
alpha-5 (via Actions), mas funciona no alpha-4 (via EAS, keystore
canônico).

Resolução planejada (Q17.e em sprint futura):
1. Exportar keystore EAS via `eas credentials --platform android` (manual via web).
2. Armazenar como base64 em GitHub Secrets (`ANDROID_KEYSTORE_BASE64`,
   `ANDROID_KEYSTORE_PASSWORD`, `ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD`).
3. Workflow decodifica + assina o APK com esse keystore. Mesmo SHA-1
   → OAuth funciona.

Por ora, outras features (Q5.1/Q5.2/Q6/Q11.a-c/Q14/Q15/**Q17 Health
Connect**/Q18/Q19) funcionam normalmente no APK alpha-5.

### Lessons learned sessão 7

- **Erros distintos a cada tentativa indicam progresso.** Pipeline
  convergiu em 6 iterações, cada uma desbloqueando o próximo
  gargalo. Build #6 foi 3-4× mais rápido que #2 e #3 porque o
  cache Gradle pegou.
- **Health Connect tem requisitos cruzados.** Pacote 3.5.0 exige
  minSdk 26 (Android 8+) E compileSdk 35 (devido a transitive deps
  `androidx.core:core-splashscreen:1.2.0-alpha02`). Documentado na
  spec Q17 mas só descoberto na prática.
- **Bash heredoc `echo >>` é frágil.** Quando o arquivo previamente
  não termina em `\n`, o `echo` cola na linha existente. Padrão
  cirúrgico: `printf '\n%s\n' "val" >> file` ou `sed`.
- **APK arm64-v8a only é prático pra alpha/beta.** 95%+ dos
  Androids modernos cobertos, 3× redução no tamanho.
